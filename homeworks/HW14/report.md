# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- В работе использовалась небольшая текстовая база знаний по теме биологии и поведения домашних кошек. Эта тема выбрана потому, что документы относятся к одной предметной области, по ним можно задавать естественные вопросы, а ответы удобно строить через retrieval и mini-RAG.
- Документы разбивались на текстовые чанки фиксированного размера по словам. В основном pipeline использовался простой и воспроизводимый способ чанкинга с overlap.
- Для векторизации был выбран один фиксированный способ построения векторных представлений: **TF-IDF fallback**. Полученные векторы использовались в retrieval-конвейере с индексом `FAISS`.
- Базовая проверка retrieval показала, что на наборе из 10 контрольных запросов система достигает `hit@3 = 0.8` и `recall@3 = 0.8`. Для большинства запросов релевантный документ находился на первом месте.
- После обновления базы знаний и переиндексации retrieval заметно улучшился на новых запросах: для специальных запросов по новым документам метрики выросли с нуля до `hit@3 = 1.0` и `recall@3 = 1.0`.
- Поверх retrieval был собран учебный mini-RAG: по запросу извлекались top-k чанков, из них собирался контекст, затем из контекста формировался ответ и возвращались источники.

## 3. База знаний и подготовка документов

- Предметная область: биология и поведение домашних кошек
- Источник документов: небольшая вручную составленная база знаний из коротких тематических текстов
- Число исходных документов: 12
- Число чанков после разбиения: 24
- Какой способ чанкинга использовался: простое разбиение документов на последовательности слов фиксированной длины
- Основные параметры чанкинга: `chunk_size = 80`, `overlap = 20`
- Комментарий (3-6 предложений): выбранная база знаний хорошо подходит для retrieval, потому что все документы относятся к одной понятной теме и содержат осмысленные факты, определения и объяснения. По таким текстам легко формулировать контрольные запросы и вручную указывать релевантные документы. Объём базы небольшой, поэтому её удобно обрабатывать в учебном ноутбуке без внешней векторной БД. При этом документы достаточно содержательны, чтобы после чанкинга получить полезные фрагменты для retrieval и mini-RAG. По такой базе знаний естественно строить pipeline: сначала искать релевантные источники, затем формировать ответ на их основе.

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

- Какая embedding-модель или способ векторизации использовались: фиксированный `TF-IDF fallback`
- Какая метрика сходства использовалась: inner product для индекса `FAISS IndexFlatIP`; retrieval выполнялся по векторным представлениям чанков
- Какой индекс `FAISS` использовался: `IndexFlatIP`
- Какой `top_k` использовался по умолчанию: `top_k = 3` для оценки retrieval и `top_k = 3` для mini-RAG
- Что показали первые примеры retrieval: на первых примерах retrieval возвращал тематически правильные документы. Например, запрос про сумеречное зрение кошек выводил на первом месте документ `cat_003`, запрос про вибриссы — документ `cat_005`, а запрос про мурлыканье — документ `cat_006`. Это показало, что retrieval-контур работает осмысленно уже на базовой проверке.

### 4.2. Контрольные запросы и оценка retrieval

- Сколько контрольных запросов использовалось: 10 
- Что считалось релевантным ответом: релевантным считался документ с нужным `doc_id`, заранее заданный для каждого запроса в benchmark
- Какие метрики retrieval считались: `hit@k`, `recall@k`, дополнительно `MRR@k`
- Итоговый `hit@k`: `hit@3 = 0.8`
- Итоговый `recall@k`: `recall@3 = 0.8`
- Дополнительные метрики (если были): `MRR@3 = 0.8`
- Какие запросы оказались самыми простыми: самыми простыми оказались вопросы, в которых ключевая тема была представлена в базе знаний явно и достаточно узко: например, про сумеречное зрение, вибриссы, мурлыканье, сон и происхождение домашней кошки. Для них релевантный документ обычно находился на первом месте.
- Какие запросы оказались самыми проблемными: самыми проблемными оказались запросы `q_06` про территориальную маркировку и `q_07` про облигатных хищников. Для них релевантный документ не попадал в top-3, и `hit@3 = 0`.

## 7. Анализ

- Выбранная база знаний оказалась удачной для учебной retrieval-задачи, потому что она компактная, тематически однородная и по ней легко формулировать контрольные вопросы. 
- Простое разбиение на чанки по словам оказалось достаточным для такого объёма данных и позволило получить 24 фрагмента, пригодных для поиска. Эксперимент с параметрами показал, что для этой небольшой базы различия между `chunk_size = 60` и `chunk_size = 90` не проявились в средних метриках. Это полезный результат сам по себе: он показывает, что на компактной базе знаний изменение одного параметра не всегда приводит к заметному изменению качества retrieval. 
- Формальная оценка retrieval через `hit@k`, `recall@k` и `MRR@k` оказалась полезной, потому что позволила не просто смотреть на отдельные примеры, а количественно оценить поведение системы. 
- Базовая оценка показала, что retrieval в целом работает хорошо, но есть проблемные запросы, для которых top-3 выдача оказывается нерелевантной. 
- Обновление базы знаний было особенно показательным этапом: после добавления трёх новых документов и переиндексации retrieval начал находить ответы на запросы, которые до этого не поддерживались исходной базой. Это хорошо демонстрирует, зачем в retrieval-системах нужна переиндексация после изменения корпуса. 
- Ошибки mini-RAG также оказались показательными: проблемы возникали либо из-за слишком широкого вопроса, либо из-за того, что база знаний не содержала прямого ответа. В некоторых случаях retrieval возвращал частично релевантные документы, но собранный контекст всё равно оставался неполным. 
- Текущее решение остаётся учебным: здесь нет сложного reranking, нет большой генеративной модели и нет внешней векторной БД. Тем не менее для учебной задачи такой pipeline достаточно хорошо демонстрирует связь между retrieval, качеством контекста и итоговым качеством ответа.

## 8. Итоговый вывод

- В качестве базового retrieval-конвейера для такой задачи я бы оставила простой воспроизводимый pipeline: фиксированный чанкинг, фиксированный TF-IDF backend, индекс `FAISS`, benchmark-запросы и оценку качества через `hit@k` и `recall@k`.  
- Главный вывод по работе состоит в том, что качество retrieval нельзя оценивать только на глаз — его нужно проверять формально и смотреть на слабые запросы. 
- Также стало понятно, что итоговое качество mini-RAG напрямую зависит от того, какие источники вернул retrieval. Даже простой генератор ответа может работать осмысленно, если retrieval находит хорошие фрагменты, и наоборот, слабый retrieval сразу ухудшает итоговый ответ.
