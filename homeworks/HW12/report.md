# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

- В работе использовался датасет S12-hw-dataset.csv, а прогнозировалось следующее значение числового ряда из колонки target.
- Данные были отсортированы по времени, после чего выполнен корректный train / validation / test split без перемешивания строк.
- Сравнивались четыре подхода: B1 (naive-last), B2 (moving-average), B3 (ridge-lag-features) и R1 (gru-forecast).
- Лучшим подходом оказался R1, так как он показал наименьшую ошибку на validation по основной метрике MAE.

Также заметна регулярная повторяющаяся структура, то есть у ряда есть сезонность, вероятно связанная с коротким периодическим циклом. ## 4. Temporal split и признаки

### 4.1. Разбиение по времени

- Как именно выполнен `train / validation / test`: данные разбиты строго по времени, без перемешивания строк: ранняя часть использовалась как train, следующая по времени часть как validation, а самый поздний участок как test.
- Какие доли или интервалы использовались: использовались доли train = 70%, validation = 15%, test = 15%.
    train: с 2025-01-01 00:00:00 по 2025-05-06 23:00:00
    validation: с 2025-05-07 00:00:00 по 2025-06-02 23:00:00
    test: с 2025-06-03 00:00:00 по 2025-06-29 23:00:00
- Почему `random split` здесь некорректен: для временного ряда модель должна обучаться на прошлом и проверяться на будущем. Random split смешивает наблюдения из разных моментов времени, создаёт риск утечки информации из будущего и даёт слишком оптимистичную оценку качества. Поэтому для forecasting корректен именно temporal split.

### 4.2. Признаки для baseline-моделей

- Какие лаги использовались: lag_1, lag_7, lag_14
- Какие rolling-признаки использовались: rolling_mean_7, rolling_std_7
- Какие календарные признаки использовались: dayofweek, month, а также циклическое кодирование dow_sin, dow_cos, month_sin, month_cos
- Как обрабатывались пропуски после построения лагов/rolling: после построения признаков начальные строки с недостаточной историей были удалены через dropna, так как для них lag- и rolling-признаки ещё не могли быть посчитаны.
- Использовалось ли масштабирование и где именно оно обучалось: для B3 использовался StandardScaler для признакового пространства, обученный только на train. Для R1 использовался StandardScaler для ряда target, также обученный только на train, после чего применён к validation и test.

## 5. Модели и эксперименты (B1, B2, B3, R1)

Опишите коротко и сопоставимо:

- B1 (`naive-last`): baseline, где прогноз на следующий шаг равен последнему известному значению ряда. В реализации это соответствует использованию признака lag_1 как прогноза.
- B2 (`moving-average`): baseline, где прогноз строится как скользящее среднее по предыдущему окну. В работе использовался признак rolling_mean_7.
- B3 (`ridge-lag-features`): линейная модель Ridge на лаговых, rolling- и календарных признаках. Это табличный baseline, который уже умеет использовать информацию о прошлом и частично учитывать сезонные эффекты.
- R1 (`gru-forecast`): рекуррентная модель GRU, обучаемая на оконном представлении ряда. На вход подаётся последовательность из нескольких последних значений, а модель предсказывает следующее значение ряда.

Дополнительно:

- Основная метрика выбора лучшей модели: best_val_mae
- Дополнительные метрики: RMSE, MAPE, sMAPE
- `window_size` для `GRU`: 24
- Batch size: 64
- Максимальное число эпох: 20
- Optimizer: Adam
- Learning rate: 1e-3
- Как сохранялась лучшая модель: лучшая версия GRU выбиралась по минимальному validation loss; соответствующий state_dict() сохранялся и восстанавливался после завершения обучения.

## 7. Анализ

Baseline B1 оказался достаточно сильным, потому что ряд обладает выраженной регулярной структурой, и последнее наблюдение действительно содержит много полезной информации о ближайшем будущем. Это типично для временных рядов с локальной инерцией и повторяющимся паттерном. Moving average (B2) сработал заметно хуже, потому что усреднение слишком сглаживает ряд и хуже ловит быстрые изменения и пики. Для этого набора данных такой baseline оказался слишком грубым.
Ridge на лаговых, rolling- и календарных признаках (B3) показала качество, очень близкое к B1. Это означает, что engineered features действительно содержат полезную информацию о структуре ряда, но линейная модель не смогла извлечь из них достаточно сложные зависимости, чтобы уверенно обойти наивный baseline. При этом B3 всё равно полезен как интерпретируемый и корректный tabular baseline без утечек.
GRU (R1) оказалась лучшей моделью. Скорее всего, это связано с тем, что она работает не с фиксированным набором признаков, а непосредственно с оконным представлением последовательности и лучше учитывает форму локальной динамики ряда. На графике прогноза видно, что модель хорошо воспроизводит общий уровень ряда и основную сезонную структуру. Однако на отдельных резких пиках и выбросах ошибки всё ещё увеличиваются, что видно и по графику остатков.
В этой задаче утечки могли бы возникнуть в нескольких местах: при использовании random split, при масштабировании на всём датасете сразу, а также при построении rolling-признаков без сдвига во времени. Поэтому важно, что здесь признаки строились только из прошлого, а scaler обучался только на train. Выбор лучшей модели по validation, а не по test, принципиален, потому что test должен оставаться честной финальной проверкой, а не участвовать в настройке решения. Наиболее показательными ошибками модели оказались промахи на экстремальных значениях и быстрых колебаниях ряда, где простой сглаженный прогноз уже не успевает точно повторять реальное движение.

## 8. Итоговый вывод

Как базовый подход для такой задачи я бы выбрала R1, потому что именно GRU показала лучший результат по validation и сохранила хорошее качество на test. При этом B1 и B3 остаются полезными baseline-моделями: они простые, быстрые и дают понятную точку отсчёта. Если ряд относительно простой и важна интерпретируемость, табличного baseline на лаговых признаках может быть достаточно. Если же ряд имеет более сложную временную структуру, как в этой работе, переход к GRU оказывается оправданным. Главный практический вывод из HW12 для меня — во временных рядах корректная валидация по времени и контроль утечек так же важны, как и сама модель.
